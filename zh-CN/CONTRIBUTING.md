## 🎯 如何贡献？

### 1. 创建新代理

有关于专业代理的想法？太棒了！添加新代理的步骤如下：

1. **Fork 本仓库**
2. **选择合适的部门**（16 个部门之一——或提议新增一个）：
   - `academic/` - 研究、学术及领域专家
   - `design/` - UX/UI 及创意专家
   - `engineering/` - 软件开发专家
   - `finance/` - 财务规划、会计及投资专家
   - `game-development/` - 游戏设计与开发专家
   - `gis/` - 地理空间、地图及空间分析专家
   - `marketing/` - 增长与营销专家
   - `paid-media/` - 付费获客与媒体专家
   - `product/` - 产品管理专家
   - `project-management/` - 项目管理及协调专家
   - `sales/` - 销售、收入及交易专家
   - `security/` - 安全架构、应用安全、渗透测试、威胁情报及事故响应
   - `spatial-computing/` - AR/VR/XR 专家
   - `specialized/` - 不适合归入其他部门的独特专家
   - `support/` - 运营及支持专家
   - `testing/` - QA 及测试专家

   > **部门由仓库根目录的 `divisions.json` 定义**——这是部门集合的唯一事实来源，由 CI 中的 `scripts/check-divisions.sh` 进行校验。
   > **提议新增一个部门**意味着：创建对应目录，在 `divisions.json` 中添加条目（label/icon/color），并在 `scripts/convert.sh` 和 `scripts/lint-agents.sh` 两者的 `AGENT_DIRS` 中添加该部门。除非这些全部一致且该目录包含至少一个代理文件，否则检查将导致构建失败。
   >
   > 注意：`strategy/`（NEXUS 操作手册/运行手册——无代理前言）和 `integrations/`（`convert.sh` 生成的按工具输出目录）**不是**部门，绝不能添加到部门列表中。

3. **按照下方模板创建代理文件**
4. **在真实场景中测试代理**
5. **提交包含你代理的 Pull Request**

## 🎨 代理设计指南

### 代理文件结构

每个代理应遵循以下结构：

```markdown
---
name: 代理名称
description: 代理专长与聚焦点的一句话描述
color: colorname 或 "#hexcode"
emoji: 🎯
vibe: 一句话人格钩子——是什么让这个代理令人难忘
services:                              # 可选——仅当代理需要外部服务时填写
  - name: 服务名称
    url: https://service-url.com
    tier: free                         # free、freemium 或 paid
---

# 代理名称

## 🧠 你的身份与记忆
- **角色**：清晰的角色描述
- **个性**：个性特质和沟通风格
- **记忆**：代理记住和学到的内容
- **经验**：领域专长与视角

## 🎯 你的核心使命
- 首要责任 1，附清晰的可交付成果
- 首要责任 2，附清晰的可交付成果
- 首要责任 3，附清晰的可交付成果
- **默认要求**：始终遵循的最佳实践

## 🚨 你必须遵守的关键规则
定义代理行事方式的、特定领域规则与约束

## 📋 你的技术交付物
代理产出的具体示例：
- 代码示例
- 模板
- 框架
- 文档

## 🔄 你的工作流程
代理遵循的分步流程：
1. 阶段 1：探索与研究
2. 阶段 2：规划与策略
3. 阶段 3：执行与实现
4. 阶段 4：审查与优化

## 💭 你的沟通风格
- 代理如何沟通
- 示例用语和模式
- 语气与方式

## 🔄 学习与记忆
代理从哪些方面学习：
- 成功的模式
- 失败的方法
- 用户反馈
- 领域演进

## 🎯 你的成功指标
可衡量的成果：
- 定量指标（带数字）
- 定性指标
- 性能基准

## 🚀 高级能力
代理掌握的进阶技术和方式
```

### 代理结构

代理文件被组织成两个语义组，映射到 OpenClaw 的工作区格式，并帮助其他工具解析你的代理：

#### 人设（代理是谁）
- **身份与记忆** — 角色、个性、背景
- **沟通风格** — 语气、话风、方式
- **关键规则** — 边界和约束

#### 操作（代理做什么）
- **核心使命** — 首要责任
- **技术交付物** — 具体产出和模板
- **工作流程** — 分步方法论
- **成功指标** — 可衡量的成果
- **高级能力** — 专业技术

无需特殊格式——只需将人设相关部分（身份、沟通、规则）与操作相关部分（使命、交付物、工作流程、指标）分开组织即可。`convert.sh` 脚本使用这些章节标题自动将代理拆分为工具特定格式。

### 代理设计原则

1. **🎭 强烈的个性**
   - 赋予代理独特的话风和性格
   - 不是"我是一个乐于助人的助手"——要具体、令人印象深刻
   - 示例："我默认找到 3-5 个问题并要求视觉证明"（证据收集员）

2. **📋 清晰的可交付成果**
   - 提供具体的代码示例
   - 包含模板和框架
   - 展示真实输出，而非模糊描述

3. **✅ 成功指标**
   - 包含具体、可衡量的指标
   - 示例："在 3G 网络下页面加载时间低于 3 秒"
   - 示例："跨账号总计 10,000+ karma"

4. **🔄 经过验证的工作流程**
   - 分步流程
   - 经过真实世界测试的方法
   - 不是理论上的——是实战检验过的

5. **💡 学习型记忆**
   - 代理能识别什么模式
   - 如何随时间改进
   - 在会话之间记住什么

### 外部服务

当外部服务（API、平台、SaaS 工具）对代理的功能至关重要时，代理可以依赖这些服务。当存在这种依赖时：

1. **在前言中使用 `services` 字段声明依赖**
2. **代理必须独立完整**——去掉 API 调用后，仍应有可用的人设、工作流程和专业知识
3. **不要复制供应商文档**——引用它们，而非复述它们。代理文件应当读起来像一个代理，而非一份入门指南
4. **优先选择有免费层的服务**，以便贡献者能测试代理

判断标准：*这个代理是为用户服务的，还是为供应商服务的？* 利用某个服务解决用户问题的代理应该放在这里。一个披着代理外衣的服务快速入门指南不该放在这里。

### 工具特定兼容性

**Qwen Code 兼容性**：代理正文支持 `${variable}` 模板语法以注入动态上下文（如 `${project_name}`、`${task_description}`）。Qwen SubAgent 使用最简前言：只需 `name` 和 `description`；`color`、`emoji` 和 `version` 字段因 Qwen 不使用而被省略。

**Codex 兼容性**：Codex 自定义代理生成为独立的 TOML 文件。Codex 集成保持最简 1:1 映射：从前言复制 `name` 和 `description`，Markdown 正文变为 `developer_instructions`。仅限源数据的元数据（如 `color`、`emoji`、`vibe` 及其他不支持的前言字段）被省略。

### 什么样才是一个出色的代理？

**出色的代理具备**：
- ✅ 窄而深的专业化
- ✅ 鲜明的个性和话风
- ✅ 具体的代码/模板示例
- ✅ 可衡量的成功指标
- ✅ 分步工作流程
- ✅ 真实世界的测试与迭代

**避免**：
- ❌ 笼统的"乐于助人的助手"个性
- ❌ 模糊的"我将帮助你......"描述
- ❌ 没有代码示例或可交付成果
- ❌ 过于宽泛的范围（万事通）
- ❌ 未经测试的理论性方法

## 🔄 Pull Request 流程

### PR 中应该包含什么（以及不该包含什么）

最容易被合并的 PR 是**一个 Markdown 文件**——一个新的或改进的代理。这是最佳路径。

对于超出此范围的，以下是保持顺畅的方式：

#### 始终欢迎作为 PR 提交
- 新增一个代理（一个 `.md` 文件）
- 改进已有代理的内容、示例或个性
- 修正拼写错误或完善文档

#### 先发起 Discussion
- 新工具、构建系统或 CI 工作流
- 架构变更（新目录、新脚本、站点生成器）
- 横跨仓库多个文件的改动
- 新增集成格式或平台

我们热爱有野心的想法——[Discussion](https://github.com/msitarzewski/agency-agents/discussions) 只是让社区在代码写出来之前有机会在方案上达成一致。这为大家节省时间，尤其是你自己的。

#### 我们始终会关闭的
- **已提交的构建产物**：生成的文件（`_site/`、编译后的资源、转换后的代理文件）不应入库。用户本地运行 `convert.sh`；所有输出均被 gitignore。
- **未经事先讨论就批量修改已有代理的 PR**——即便是善意的重新格式化也可能给其他贡献者造成合并冲突。
- **近似重复的"翻版"**：对已有代理做查找替换（例如仅替换国家或平台名称）而成的新代理，而非真正全新的专家。提交前运行 `scripts/check-agent-originality.sh`——CI 会自动运行它。

### 提交前

1. **测试你的代理**：在真实场景中使用，根据反馈迭代
2. **遵循模板**：匹配已有代理的结构
3. **添加示例**：包含至少 2-3 个代码/模板示例
4. **定义指标**：包含具体、可衡量的成功标准
5. **校对**：检查拼写错误、格式问题、表达清晰度
6. **检查原创性**：运行 `./scripts/check-agent-originality.sh path/to/your-agent.md`。它将你的代理与整个阵容对比，标记近似重复（仅替换国家/平台名称骗不了它）。新代理应该是真正全新的——如果你在为某个市场做本地化，确保平台、策略和示例确实不同，而非简单查找替换。

### 提交你的 PR

1. **Fork** 仓库
2. **创建分支**：`git checkout -b add-agent-name`
3. **做出你的改动**：添加你的代理文件
4. **提交**：`git commit -m "Add [代理名称] specialist"`
5. **推送**：`git push origin add-agent-name`
6. **发起 Pull Request**，附带：
   - 清晰的标题："Add [代理名称] - [分类]"
   - 代理做什么的描述
   - 为什么需要这个代理（使用场景）
   - 你已完成的任何测试

### PR 审查流程

1. **社区审查**：其他贡献者可能提供反馈
2. **迭代**：回应反馈并做出改进
3. **批准**：维护者在准备就绪时批准
4. **合并**：你的贡献成为 The Agency 的一部分！

### PR 模板

```markdown
## 代理信息
**代理名称**：[名称]
**分类**：[engineering/design/marketing/等]
**专长**：[一句话描述]

## 动机
[为什么需要这个代理？它填补了什么空白？]

## 测试
[你有如何测试过这个代理？真实世界使用场景？]

## 检查清单
- [ ] 原创 — 非近似重复（已运行 `scripts/check-agent-originality.sh`）
- [ ] 遵循代理模板结构
- [ ] 包含个性与话风
- [ ] 有具体的代码/模板示例
- [ ] 定义了成功指标
- [ ] 包含分步工作流程
- [ ] 已校对且格式正确
- [ ] 在真实场景中测试过
```

## 📐 风格指南

### 写作风格

- **要具体**："将页面加载时间减少 60%"而非"让它更快"
- **要具体（Concrete）**："用 TypeScript 创建 React 组件"而非"构建 UI"
- **要有记忆点**：赋予代理个性，而非空洞的企业用语
- **要实用**：包含真实代码，而非伪代码

### 格式

- 统一使用 **Markdown 格式**
- 章节标题使用 **emoji**（便于快速浏览）
- 所有代码示例使用**代码块**并标注正确的语法高亮
- 使用**表格**对比选项或展示指标
- 使用**粗体**强调，`代码格式`标注技术术语

### 代码示例

```markdown
## 示例代码块

\`\`\`typescript
// 始终包含：
// 1. 用于语法高亮的语言标注
// 2. 解释关键概念的注释
// 3. 真实、可运行的代码（非伪代码）
// 4. 现代最佳实践

interface AgentExample {
  name: string;
  specialty: string;
  deliverables: string[];
}
\`\`\`
```

### 语气

- **专业但平易近人**：不过于正式或随意
- **自信但不傲慢**："这是最佳方式"而非"也许你可以试试..."
- **有帮助但不包办代替**：假定对方有能力，提供深度
- **个性驱动**：每个代理应该有独特的话风
